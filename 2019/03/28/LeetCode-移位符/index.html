LeetCode--移位符

不得不说，自己的算法是真的差，差的也不仅仅是算法，更多的还是自己上课期间忽略的基础知识。
LeetCode第29题，题目描述如下：

给定两个整数，被除数 dividend 和除数 divisor。将两数相除，要求不使用乘法、除法和 mod 运算符。
返回被除数 dividend 除以除数 divisor 得到的商。

示例 1:
输入: dividend = 10, divisor = 3
输出: 3
示例 2:
输入: dividend = 7, divisor = -3
输出: -2

说明:
被除数和除数均为 32 位有符号整数。
除数不为 0。
假设我们的环境只能存储 32 位有符号整数，其数值范围是 [−231, 231 − 1]。本题中，如果除法结果溢出，则返回 231 − 1。

看到这道题，我的第一个想法：“一个数除以另一个数，可以转化成多次减法运算”。
然后我的第一个骚操作来了：

现在看到这段代码，我自己都佩服我自己。（手动鼓掌）

其实这个想法还是没错的，错在于没考虑极端情况

举例子说明，比如第一组测试样例，10除以3这种“正常数字”（日常接触得到的计算量级）来说，这段代码并没有问题，可以正常运行，花费的时间也在可以接受的范围之内，但是现在问题来了：极端情况下会怎么样呢？

我们给dividend取int的最大值（2的31次方-1），divisor取值1，执行这段代码需要耗时多久？
不难算出，while循环需要执行2的31次方-1次，也就是二十一亿多次，很明显时间上是不允许的。所以我们需要改进这个算法。

在百度的帮助下，我认识了一个符号，叫移位符。好吧，这可能真的是很基础的东西，但是原谅我的菜，我真的是第一次听说这个东西。
简单介绍下移位符的作用：

移位运算符在程序设计中，是位操作运算符的一种。移位运算符可以在二进制的基础上对数字进行平移。按照平移的方向和填充数字的规则分为三种：<<(左移)、>>(带符号右移)和>>>(无符号右移)。

拿右移>>举例子， 7>>1 相当于 7/2 7>>2相当于 7/4，即右移n位相当于除以2的n次方。
（从结果上看是这种效果）

具体的运算例子 11>>2

11的二进制形式为：0000 0000 0000 0000 0000 0000 0000 1011，然后把低位的最后两个数字移出，因为该数字是正数，所以在高位补零。则得到的最终结果是0000 0000 0000 0000 0000 0000 0000 0010。转换为十进制是2。

有了这个积累，我们再来看这道题。
我们现在需要做的事情，就是减少while循环的次数，怎么减少呢？当然就是每次循环多减一点啦！
先上代码：

好了，现在我来解释下这个程序。

首先看内层循环 while(m > (temp<<1))里面的东西，temp等于除数，当m（被除数的绝对值）大于两倍的除数的时候，temp乘以二，count乘以二，循环到temp的两倍大于被除数m为止。这样循环的结果是循环次数会以指数级减少。
我们还是拿最开始的例子，dividend = pow（2，31）-1；divisor = 1；
第一次进入外层循环，然后进入内层循环，内层的while执行30次后结束，此时temp = 2的30次方，count = 2的30次方。接着向下执行第18和第19两行代码，整个循环第一次结束后，m = 1。

对比21亿次循环，我们的算法有了质的飞跃（为自己鼓掌）。

但是，我的骚操作远没有结束上面放出的这段代码是我最后一次提交成功的代码。

前面还有十几次失败经验还没总结呢。

首先第一个问题，变量声明里面的 long 为什么要用long？
还是上面那个极端的例子，在判断的时候 temp<<1，如果用int，会发生超出int最大值的问题，然后temp会变成一个负数，然后这个循环永远结束不了，然后就超时了。。。。。。

第二个是边界值问题，在这个题目中，边界情况下只有一种可能会越界，就是当被除数为int最小值，除数为-1时，得到的商会大于int的最大值，所以我们需要考虑这种情况。

最后就是符号判别问题，用异或判断被除数与除数是否相同，保证输出结果符号的正确性。（当然不想用异或运算的也可以使用if写明条件判断）

当然按照这个思路做完以后，我也有个问题：为什么非要用移位运算符呢？
count = count + count不也一样么？
抱着这个疑问，我又改了改程序，又跑了一次。

结论是：确实可以，只不过会比移位运算符慢2ms

至于为什么会慢2ms…我给自己留个坑吧，初步猜测是计算机处理二进制会更快一些，当然这只是猜测而已，具体什么原因我有空再去验证一下……
